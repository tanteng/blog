AI时代，数据丰富，但高质量的上下文却难以获得。

当大家都在讨论RAG、Agent、微调时，火山引擎悄悄开源了一个底层基础设施——**OpenViking**，一个专为AI智能体设计的**上下文数据库**。

它要解决的，是一个更根本的问题：**智能体的"记忆"该怎么管理？**

> GitHub：https://github.com/volcengine/OpenViking

---

## 智能体开发面临的困境

在构建AI智能体时，开发者经常遇到这些痛点：

| 困境 | 描述 |
|-----|------|
| **上下文碎片化** | 记忆在代码中，资源在向量数据库中，技能分散在各处 |
| **上下文需求激增** | 长运行任务每次都产生上下文，截断或压缩会导致信息丢失 |
| **检索效果不佳** | 传统RAG使用扁平化存储，缺乏全局视图 |
| **上下文不可观察** | 隐式检索链像黑盒，出错时难以调试 |
| **记忆迭代有限** | 当前记忆只是交互记录，缺乏任务相关记忆 |

---

## OpenViking 核心思路

传统RAG是**碎片化的向量存储**，OpenViking提出了一个全新的范式：**"文件系统"**。

就像你管理本地文件一样管理智能体的大脑：

```
viking://
├── resources/     # 资源：文档、代码、网页
├── user/         # 用户：偏好、习惯
└── agent/        # 智能体：技能、指令、任务记忆
```

---

## 五大核心概念

### 1. 文件系统管理范式

不再将上下文视为扁平的文本切片，而是统一到虚拟文件系统。

每个上下文都有唯一的URI（如 `viking://resources/my_project/docs/api.md`），智能体可以通过 `ls`、`find` 等命令精确操作。

### 2. 分层上下文加载（L0/L1/L2）

一次性塞入大量上下文不仅贵，还容易超出窗口。

OpenViking自动将上下文处理为三层：

- **L0（摘要）**：一句话摘要，~100 tokens，用于快速检索
- **L1（概览）**：核心信息，~2k tokens，用于规划阶段
- **L2（详情）**：完整原文，按需加载

### 3. 目录递归检索

传统向量检索是"一把梭"，OpenViking设计了**目录递归检索**：

1. 意图分析 → 生成多个检索条件
2. 初始定位 → 向量检索定位高分目录
3. 精细探索 → 目录内二次检索
4. 递归深入 → 逐层探索子目录
5. 结果聚合 → 返回最相关上下文

### 4. 可视化检索轨迹

每次检索的目录浏览和文件定位轨迹都被完整保留。

用户可以清晰观察问题的根源，而不是对着一个黑盒抓狂。

### 5. 自动会话管理

内置记忆自迭代循环。会话结束后，系统自动分析任务执行结果，更新用户偏好和智能体经验。

智能体越用越聪明。

## 实验数据

OpenViking与OpenClaw集成后的效果测试：

| 实验组 | 任务完成率 | 输入Token成本 |

**结论**：
- 任务完成率提升 **43%**
- 输入Token成本降低 **91%**

## 总结

OpenViking要做的，是为智能体定义一个**极简的上下文交互范式**。

它不追求颠覆RAG，而是在RAG之上加了一层"文件系统"，让智能体的记忆管理变得可控、可观察、可迭代。

对于正在构建AI应用的开发者来说，这是一个值得关注的基础设施。
