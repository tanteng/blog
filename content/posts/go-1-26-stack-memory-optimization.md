Go 语言以高效的垃圾回收（GC）著称，但在追求极致性能的路上，内存分配始终是绕不开的话题。2026年2月发布的 **Go 1.26** 带来了一个重要的编译器优化：**现在可以在更多情况下将 slice 的后备存储（backing store）分配在栈上，而不是堆上**。

这意味着什么？想象一下，当你在函数内创建一个 slice 时，如果它不会逃逸出函数作用域，Go 1.26 会直接把它放在栈上，无需经过堆分配。这不仅减少了 GC 压力，还提升了缓存局部性，是一个"免费"的性能提升。本文将深入讲解栈与堆的区别、逃逸分析的原理，以及如何利用这个新特性。

## 栈 vs 堆：物理存储

| 栈 (Stack) | 堆 (Heap) |
|------------|-----------|
| 函数局部变量 | 动态分配 |
| 自动分配/释放 | 需GC回收 |
| 速度快 | 速度稍慢 |

## Go 1.26 的栈内存优化

Go 1.26（2026年2月发布）带来了一个重要的优化：**编译器现在可以在更多情况下将 slice 的后备存储（backing store）分配在栈上，而不是堆上**。

### 好处

- 减少堆分配，降低 GC 压力
- 提升缓存局部性（cache locality）
- 性能免费提升

## 例子分析

```go
func process() []int {
    s := []int{1, 2, 3, 4, 5}  // 分配在哪？
    return s                    // 返回会发生什么？
}
```

### 情况1：不返回

```go
func process() {
    s := []int{1, 2, 3, 4, 5}
    sum := 0
    for _, v := range s {
        sum += v
    }
    println(sum)
}
```

→ **栈分配** ✅

编译器通过**逃逸分析**确定 `s` 只在函数内使用，不会逃逸，直接放栈上。

### 情况2：返回 s

```go
func process() []int {
    s := []int{1, 2, 3, 4, 5}
    return s  // 调用方需要这个 slice！
}
```

→ **可能逃逸到堆** ⚠️

因为 `return s` 把 slice 传给调用者了，调用者可能在函数返回后继续使用。所以编译器保守起见，会把它放堆里。

## 逃逸分析核心逻辑

Go 编译器在编译时做**静态分析**，判断一个变量：

| 条件 | 结果 |
|------|------|
| 传给返回值 | **逃逸** → 堆 |
| 传给指针/interface | **逃逸** → 堆 |
| 闭包引用 | **逃逸** → 堆 |
| 变量大小不确定（make without cap） | 可能逃逸 |
| 仅在函数内使用 | **栈分配** ✅ |

## 实战：看你的代码是否逃逸

```bash
# -m = 打印优化决策
# -l = 禁用内联（看得更清楚）
go build -gcflags="-m -l" your.go
```

输出示例：

```
./your.go:10:6: can inline process
./your.go:11:10: inlining call to make
./your.go:12:6: process() -> s does not escape   # 栈分配！
./your.go:20:6: leaking param: s                # 逃逸！
```

## 一句话总结

> **Go 1.26 让编译器更大胆地先把 slice 放栈上**，如果后续分析发现不需要逃逸，就省了一次堆分配。只有真正需要时才搬到堆。

这样既安全（不会 use-after-free），又高效（减少不必要的堆分配）。
