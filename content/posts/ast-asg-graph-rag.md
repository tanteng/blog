你是否遇到过这样的情况？问AI关于代码的问题，它总是"一本正经地胡说八道"，给出的答案看似有理，实则牛头不对马嘴。

传统RAG（检索增强生成）虽然能解决大模型"知识陈旧"的问题，但对于代码这种高度结构化的知识，仍然力不从心。

## 传统RAG的痛点

在深入之前，我们先理清一个问题：**为什么传统RAG处理代码效果不好？**

传统RAG的工作流程是：
```
用户问题 → 检索文档 → 塞给LLM → 生成答案
```

问题在于：**代码不是普通文档**。

代码有复杂的结构——函数调用关系、变量类型、依赖链路。这些语义信息在普通文本检索中全部丢失了。

举个例子：
- 问："这个函数在哪里被调用了？"
- 传统RAG：全文搜索"调用"，找到一堆包含关键词的片段，但无法回答"在哪里"
- 理想答案：直接给出调用关系图

## 核心概念：AST与ASG

这个项目引入了两个概念来结构化代码：**AST** 和 **ASG**。

### AST（抽象语法树）

代码的**语法结构**树状表示，关注"怎么写"。

```python
def add(a, b):
    return a + b
```

AST表示：
```
FunctionDef
  name: add
  args: [a, b]
  body:
    Return
      BinOp (+)  ← 左(a) + 右(b)
```

**特点**：树状结构，关注语法，易于生成

### ASG（抽象语义图）

代码的**语义结构**图状表示，关注"什么意思"。

```python
def add(a, b):
    return a + b

c = add(1, 2)
```

ASG表示：
```
Function: add
  Parameter: a ──┐
  Parameter: b ──┼──> Returns: a + b
                │
Variable: c
  Value: add(1, 2) ──> 调用 add 函数
```

**特点**：图状结构，能表示复杂关系，需要类型推断

---

## AST vs ASG

| | AST | ASG |
|---|---|---|
| 结构 | 树 | 图 |
| 关注点 | 语法 | 语义 |
| 生成难度 | 易（标准解析） | 难（需类型推断） |
| 用途 | 代码重构、静态分析 | 程序理解、合成、验证 |

## 核心思路：Graph RAG

项目的核心思路非常清晰：

```
代码 → AST/ASG → 提取关系(函数调用、变量类型、依赖) → 图数据库 → Graph RAG 问答
```

**三步走：**

1. **结构化提取**：用AST/ASG解析代码，提取变量、函数、类型的关系
2. **图数据库存储**：将提取的关系存入图数据库
3. **语义检索**：根据语义关系检索，而非关键词匹配

---

## 这样做有什么好处？

| 能力 | 说明 |
|-----|------|
| **更准确** | 提取变量、函数、类型的**关系**供RAG检索 |
| **更高效** | 按语义结构索引，快速定位代码关系 |
| **更可扩展** | 自动提取+索引大型代码库 |

**应用场景：**

- 问："变量 c 在函数调用 add 后的值是什么？" → 直接通过图关系回答
- 问："这个API被哪些服务调用？" → 遍历调用图
- 问："修改这个函数会影响哪些地方？" → 分析依赖链路

---

## 总结

AST-ASG-Graph-RAG 的本质是：**让AI真正"理解"代码结构**，而不仅仅是匹配关键词。

当其他RAG还在用"全文检索"时，Graph RAG已经用"知识图谱"理解代码了。

这也是未来的趋势：**结构化数据 + 大模型 = 更智能的代码助手**。
